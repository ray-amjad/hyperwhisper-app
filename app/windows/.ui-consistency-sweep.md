# Windows ↔ macOS Parity Sweep

Canonical macOS design SPEC translated to the existing WPF token vocabulary, plus a Windows→macOS page MAP. This is the yardstick for auditing each Windows page on **typography, spacing, cards, controls, and colors**. Width is **LOCKED** and out of scope (see policy below).

Sources:
- macOS tokens: `app/macos/hyperwhisper/Utilities/DesignConstants.swift`, `app/macos/hyperwhisper/Views/Settings/SettingsSharedStyles.swift`
- WPF vocabulary: `app/windows/HyperWhisper/Themes/Generic.xaml`, `app/windows/HyperWhisper/Themes/Brushes.xaml`

---

## SPEC — macOS design intent → concrete WPF targets

macOS points map ~1:1 to WPF DIPs. macOS semantic fonts resolve to explicit `FontSize`/`FontWeight`.

### Typography (by role)

| Role | macOS | WPF target |
|------|-------|-----------|
| Page title | `Font.largeTitle.weight(.bold)` (`Typography.pageTitle`) | FontSize ~26, FontWeight Bold, `TextPrimaryBrush` |
| Section title | `Font.title.weight(.semibold)` (`SettingsSection`, `Typography.sectionTitle`) | FontSize ~20, FontWeight SemiBold, `TextPrimaryBrush` |
| Group header | `Font.title3.weight(.semibold)` (`SettingsGroupHeader`) | FontSize 15–16, FontWeight SemiBold, `TextPrimaryBrush` |
| Row title / headline | `Font.headline` (`Typography.rowTitle`) | FontSize ~14, FontWeight SemiBold, `TextPrimaryBrush` |
| Body / subtitle | `Font.body` (`Typography.subtitle`) | FontSize 13, FontWeight Normal, `TextPrimaryBrush` |
| Caption / secondary | `Font.caption` / `.system(size:12)` (`Typography.caption`, `.small`) | FontSize 11–12, `TextSecondaryBrush` |
| Button | `.system(size:14,weight:.medium)` (`Typography.button`) | FontSize 13–14, FontWeight SemiBold/Medium (matches `PrimaryButtonStyle`/`MacButtonStyle`) |

Subtitle/secondary text is always `TextSecondaryBrush`. Inline "Learn more" link text is the accent color (`AccentBrush`), not hardcoded blue.

### Spacing scale (macOS `DesignConstants.Spacing` → WPF DIPs)

| Token | macOS | WPF |
|-------|-------|-----|
| xs | 4 | 4 — minimal gaps (icon↔status, badge insets) |
| small | 8 | 8 — compact gaps, card horizontal inset |
| medium | 12 | 12 — standard gap between elements, inter-card gap |
| large | 20 | 20 — major section breaks (`SettingsSection` internal spacing) |
| xl | 24 | 24 — main content padding (`PagePadding` = 24,24) |
| rowPadding | 10 | 10 — internal padding of settings rows |
| sectionSpacing | 20 | 20 — gap between major sections |
| header | H24 / V20 | header block padding 24 horizontal, 20 vertical |

Reference views: HomeView content padding `24`, inter-card spacing `12–16`; ModelLibrary content padding `V24/H28`, stack spacing `14`. Page content uses `PagePadding` (24,24); cards stack with ~12 gaps; sections separated by ~20.

### Card style

macOS `.thinMaterial` fill + `separatorColor` 1px stroke + continuous corner radius 10 (`CardStyle`, `SettingsCard`, `SettingsRowContainerStyle`). →

WPF `Style x:Key="CardStyle"`:
- Background `CardBackgroundBrush`
- BorderBrush `BorderBrush`, BorderThickness `1`
- CornerRadius `{StaticResource CardCornerRadius}` (9; macOS uses 10 — both acceptable, prefer the shared token)
- Padding `16` (macOS rows pad 10–12 internally; section cards 12)

Inside-card row dividers: macOS uses `Divider()` between stacked rows → WPF 1px `BorderBrush` separator (full width or inset to row content). Inline alert/prompt banners (e.g. Home API-key prompt) use a tinted rounded rect; on Windows use semantic brushes (`WarningBackgroundBrush`/`WarningBorderBrush`, `InfoBackgroundBrush`, etc.) — never hardcoded orange/blue.

### Control patterns

- **Buttons:** prominent/primary action → `PrimaryButtonStyle` (AccentBrush, TextOnAccentBrush, FontWeight SemiBold, radius 7). Secondary/default → `MacButtonStyle` (InputBackgroundBrush, 1px BorderBrush). macOS `.borderedProminent` → Primary; `.bordered`/plain → Mac.
- **List / table rows:** macOS `LazyVStack` of rows separated by `Divider()` inside a card (ModelLibrary table, settings cards). WPF: ItemsControl/ListView templated rows inside a `CardStyle` border, 1px `BorderBrush` separators, row padding ~10–12. ListView/ListBox/DataGrid MUST set `ScrollViewer.HorizontalScrollBarVisibility="Disabled"` so text wraps instead of overflowing.
- **Search pill + filter menu:** macOS search bar = rounded rect (radius 10) `primary.opacity(0.05)` fill + magnifier glyph + inline removable `FilterChip`s + trailing filter Menu. WPF: rounded search container using `InputBackgroundBrush`/`CardBackgroundBrush` + `BorderBrush`; filter chips with `BadgeGray*` or `SelectedBackgroundBrush`; filter menu = themed ComboBox/ContextMenu.
- **Group headers between cards:** macOS `SettingsGroupHeader` (title3 semibold, 4pt leading inset). WPF: TextBlock FontSize 15–16 SemiBold, `TextPrimaryBrush`, small leading margin.
- **Empty states:** macOS centered secondary text ("No models match your filters", padding V40). WPF: centered `TextSecondaryBrush` TextBlock with generous vertical padding.
- **Combo / TextBox / CheckBox:** use the global themed styles in `Generic.xaml` (radius 8 inputs, 7 buttons) — do not re-template per page.

### Color rule (HARD)

All color comes from `DynamicResource` theme brushes defined in `Brushes.xaml`:
`TextPrimaryBrush`, `TextSecondaryBrush`, `TextTertiaryBrush`, `TextOnAccentBrush`, `CardBackgroundBrush`, `ContentBackgroundBrush`, `SidebarBackgroundBrush`, `BorderBrush`, `AccentBrush`/`AccentHoverBrush`, `HoverBackgroundBrush`, `SelectedBackgroundBrush`, `InputBackgroundBrush`, and semantic `Success*`/`Warning*`/`Error*`/`Info*`/`Badge*`/`Statistics*` brushes. **No hardcoded hex** anywhere. macOS `.secondary` → `TextSecondaryBrush`; macOS accent/`.blue` link → `AccentBrush`; macOS green/orange/red status → `SuccessBrush`/`WarningBrush`/`ErrorBrush`.

## WIDTH POLICY (LOCKED)

Every Windows content page intentionally uses `MaxWidth = ContentMaxWidthWide` (720). **Do not propose or make any content max-width changes.** The only width/layout issue in scope is a genuine horizontal-**overflow bug** (e.g. a ListView/ListBox/DataGrid measuring wider than its column because `ScrollViewer.HorizontalScrollBarVisibility` is not `Disabled`, so text does not wrap). `SettingsPage` (sidebar shell) and `HistoryPage` (split) have intentional structure — only typography/spacing/color/control-token issues apply, not layout restructure. `Views/Windows/*` (dialogs/toasts/overlays) and `Views/Controls/*` (reusable controls) are audited only for color-token and typography/spacing-token consistency, not page-layout rules.
